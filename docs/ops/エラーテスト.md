# エラーから学んだルール集

セッション開始時に必ず読むこと。同じミスを繰り返さないための蓄積ファイル。

---

## カテゴリ1: 本番DB vs 開発DB

### ルール: 本番DBと開発DBは完全に別のデータベース
- **ミスパターン**: executeSqlやcode_executionでデータを投入して「完了」と思い込む
- **正しい手順**:
  1. 開発DBにデータ投入
  2. `server/seed.ts` の `runSeed()` にシードコード追加（永続化）
  3. 本番curl経由で即時投入（`prod-data-ops`スキル参照）
  4. 本番デプロイログで `[Seed]` 確認
  5. 本番で動作確認
- **完了の定義**: ユーザーがアクセスする本番環境（d-planet.replit.app）で正しく動作している状態

---

## カテゴリ2: テスト実行

### ルール: テストアカウント情報を毎回確認する
- **ミスパターン**: ログインパスやアカウント情報を間違えてe2eテストが失敗
- **正しい手順**: replit.mdの「テストアカウント」セクションを確認してからテスト実行
- **テストアカウント情報**:
  - 管理者: `D-Planet管理者` / `admin2025`
  - テストユーザー: ゼノ・クオーツ / `xeno@d-planet.local` / `xeno2026`
  - ログインパス: `/login`
  - data-testid: `input-username`, `input-password`, `button-login`

### ルール: ログインが必要なページのテストはログイン手順を含める
- **ミスパターン**: ログインなしでページにアクセスしてテスト失敗
- **正しい手順**: テストプランの最初にログイン手順を入れる

---

## カテゴリ3: ユーザー仕様の尊重

### ルール: ユーザーが決定した仕様を勝手に変更しない
- **ミスパターン**: モデル数や機能範囲を効率化のために勝手に縮小
- **対策**: 変更前に必ずユーザーに確認。dev_recordsで過去の決定を検証

### ルール: マークアップ率などの内部数値をUI画面に表示しない
- **ミスパターン**: 開発者向けの原価情報をユーザー向け画面に出す
- **対策**: ユーザー向けUIには「月777往復=¥3,690」のようなユーザー目線の表現のみ

### ルール: 「軽量」「コスパ」等の表現を使わない
- **対策**: モデルの個性や特徴で表現する

---

## カテゴリ4: コード変更

### ルール: 実装前にプランをユーザーに提示して承認を得る
- **ミスパターン**: いきなりコードを書き始める
- **正しい手順**: 3ステップ以上の作業は必ずプランモードでタスク一覧を提示してから着手

### ルール: 影響範囲を最小限にする
- **ミスパターン**: 関連ないファイルまで巻き込んで変更
- **対策**: 変更は必要な箇所のみ。新たなバグの混入を避ける
