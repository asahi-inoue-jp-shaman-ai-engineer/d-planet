# SESSION_RULES — エージェント運用ルール

このファイルは **毎セッション開始時に必ず読むこと**（replit.mdに指示あり）。

---

## 1. セッション開始時チェックリスト

新しいセッションを開始したら、以下を **必ず順番に** 実行する:

1. replit.mdを読む（自動で読み込まれる）
2. **このSESSION_RULES.mdを読む**
3. **dev_recordsのactiveレコードを確認する** — `SELECT * FROM dev_records WHERE status = 'active' ORDER BY priority DESC, updated_at DESC;` を実行し、現在有効な決定事項を把握する
4. **agent_session_contextから前回の文脈を復元する** — `SELECT * FROM agent_session_context ORDER BY created_at DESC LIMIT 1;` で直近セッションの作業状態を確認する
5. 前回の未完了タスクがあれば、ユーザーに報告してから作業を開始する

---

## 2. ユーザー発言の取り扱いルール

- ユーザーの発言は **原文のまま** 記録・参照する。勝手に要約・解釈・意訳しない
- 仕様に関する発言は **dev_recordsに保存** する（category: 'decision' or 'specification'）
- 曖昧な指示は **必ず確認してから進める**（勝手に解釈しない）
- ユーザーが「こうしたい」と言ったら、**それが仕様**。反対意見があれば実行前に言う
- ユーザーの過去の決定を覆す場合は、**dev_recordsの該当レコードを提示** して確認を取る

---

## 3. 変更前の必須確認事項

以下の変更を行う前は、**必ずdev_recordsを検索し、ユーザーに確認を取る**:

| 変更内容 | 必須アクション |
|---------|-------------|
| モデル数・モデル選定の変更 | dev_recordsで過去の決定を検索 → ユーザーに確認 |
| 既存機能の削除・縮小 | **絶対にユーザー承認なしでやらない** |
| APIの破壊的変更 | 影響範囲を洗い出してからユーザーに報告 |
| 料金体系・マークアップの変更 | dev_recordsの該当レコードを確認 → ユーザーに確認 |
| UIの大幅な変更 | 変更前後の差分をユーザーに説明 |

**禁止事項:**
- ユーザーが決定した仕様を勝手に縮小・変更すること
- マークアップ率などの内部数値をユーザー向け画面に表示すること
- dev_recordsに記録された決定を確認せずに矛盾する実装をすること

## 4. エラー防止ルール

replit.mdの「過去のバグから学んだルール」セクションと連動:

- **新しいバグパターンを発見したら**、replit.mdの「過去のバグから学んだルール」に即追記する
- **SQL配列系のバグパターン** は特に警戒:
  - `ANY(${array})` は **絶対禁止** → `sql.join` + `IN` か `inArray()` を使う
- **ポート競合** — EADDRINUSE が出たら `fuser -k 5000/tcp` で解消

### デプロイ前チェックリスト
デプロイする前に、以下を **すべて** 実行する:

1. 開発環境でcurlを使い、今回変更した全APIエンドポイントの動作確認
2. fetch_deployment_logsで本番の現在の状態を確認する
3. SQL配列系パターンの再チェック — `ANY(${array})` が混入していないかgrepで確認
4. 開発DBで変更したマスタデータ（モデルリスト、料金、シードデータ等）が本番にも必要か確認し、必要なら本番用のAPI呼び出しやシード処理を準備する

### デプロイ後チェックリスト
デプロイ完了後に、以下を **すべて** 実行する:

1. 本番APIをcurlで叩いて動作確認する（認証が必要なら管理者セッションで）
2. fetch_deployment_logsでエラーが出ていないか確認する
3. 開発DBで削除・変更したデータが本番でも対応済みか確認する
4. ユーザーに本番動作確認の結果を報告する

### 開発DB vs 本番DBの乖離対策
- 開発DBと本番DBは **完全に別環境**。開発DBでの変更は本番に一切反映されない
- マスタデータの修正（モデルリスト、料金設定、シードデータ等）はデプロイ後に **本番APIでも同じ操作を実行** すること
- デプロイ前に **fetch_deployment_logsで本番の現状を把握** する習慣をつける
- 空データ・不整合データが本番に残っていないか、デプロイ後に必ず確認する

---

## 5. セッション終了時チェックリスト

セッションを終える前に、以下を実行する:

1. **agent_session_contextに保存** — 今セッションの作業内容・未完了タスク・次のステップをDBに記録
2. **dev_recordsに保存** — 今セッションで行った重要な決定事項をDBに記録
3. **replit.mdを更新** — 新しいバグパターンや仕様変更があれば反映
4. **未完了タスクを明記** — 次セッションで引き継げるようにする
